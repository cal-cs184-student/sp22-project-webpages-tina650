Overview

For this project, we implemented cloth simulation with self-collision properties that demonstrates different behaviors under different density, spring, and damping conditions. We also implemented shaders that were applied to the models in each of our scenes and simulations to enhance the quality of our scene’s models. This project was another interesting and satisfying endeavor to explore and take on as we were well aware of the importance of cloth simulation to make graphical products like CGI or video games look more realistic, but we were not aware of how these worked until the relevant lectures and the different implementations we had to do for this. It was interesting to see how concepts in physics like springs and forces played a part in affecting different areas of the cloth plane under contact with varying objects and under different pinned conditions.

Part 1: Masses and Springs

In order to build the grid first we added all the point masses. We do 2 nested for loops to transverse them in row-major order. For each space in the grid we checked if the orientation was vertical or horizontal. If horizontal then we just set the position with the y coordinate to 1, and if it was vertical we set the z to a randomly generated offset. Then we emplace back each point mass to its position and pinned as false. After we have added all the point masses, we then for loop through the pinned masses and set the point mass at the index of all pinned masses pinned to true. We made sure to convert the 2D index into the index for a 1D vector. After that we add the springs by doing another row-major double for loop. Then we checked certain conditions to see what kind of constraint to set the spring type to.

Part 2: Simulation via numerical integration

We first calculated the external forces acting on the cloth by adding up F = ma for each “a” value in external_accelerations, which we then applied to each of the point masses. For each spring in the cloth plane, we also applied a force with magnitude ks * 0.2 * ( || point mass B’s position - point mass A’s position || - rest length) where A and B are the point masses at the ends of the spring along the spring’s axis. To make sure that the position of each point mass moves under these applied forces, we used Verlet integration to calculate the new position (and update the last position to what was the current position). The new position was calculated by the Verlet integration expression: positiont+dt = positiont + (1 - cp->damping / 100) * (positiont+dt - positiont) + at * delta_t2. To prevent too much deformation, we also add constraints for each spring such that the length of the spring never exceeds 1.1 * its rest length. For the calculations made for the springs, we made sure they were done only if its spring type(s) were enabled.

Part 3: Handling collisions with other objects

For sphere collision, we first calculated the distance between the sphere’s origin and the point mass’s position. Then we checked if this distance normalized is smaller than or equal to the radius of the sphere because it is, then the point mass either intersects with or is inside the sphere. If that is true we find the tangent by finding the unit vector of the point mass’s position minus the origin of the sphere multiplied by the radius and then adding the origin. We also found the correction vector by subtracting the point mass’s last position from the tangent. Lastly, we update the point mass’s position to the point mass’s last position plus the correction vector scaled by 1 - friction. Also, we made sure to update simulate for it to handle collisions with other primitives by double for looping through the point masses and collection objects and calling collide on each collision object with each point mass.  

For the plane collision, we first calculated and checked the distance between the point mass’s position and the plane as well as the point mass’s last position and the plane (which can be calculated by the dot product of the  to check whether or not it crosses through the plane. If it does (i.e. if the sign of the two distances are opposite, meaning if the two positions are on opposite sides of the plane), then, we have to add a correction vector to the last position such that the current position is a displacement of value SURFACE_OFFSET above the plane (on the side of the last_position). This took a lot of debugging to implement as we ran into issues of the fallen plane flickering and been visible from both sides of the ground plane, but through some visualization of the vectors, we understood that the correction vector would need to be the vector added to the last_position such that the resulting position vector equals the tangent point + SURFACE_OFFSET * the normal vector * the direction of the plane that last_position resides in. The tangent point would be calculated by the point mass’s position + the plane’s normal vector * the signed distance to the plane.

Part 4: Handling self-collisions

For this part, we had to implement self-collision in which the cloth folds in on itself without piercing through and ignoring the contact with its own surface. To do so, we first had to implement a hashing function for splitting the cloth into boxes of size w x h x t where w = 3.0 * width of the cloth / num_width_points, h = 3.0 * height of the cloth / num_height_points, and t = the max of w and h. Thinking about how indices for 2-D matrices implemented as a list or a pointer worked by taking (row, column) values from prior projects (where index i = row * width + column), we took this concept into the 3-D by doing the same thing for the 2-D plane but adding the third dimensional value z multiplied by width * height (or the max value for each stacked plane for each z value we are skipping through to get to our coordinate/box). Therefore the hash value would then equal y * width + x + z * width * height; Next, we implemented the function build_spatial_map in which we built the hash table for grouping the point masses into their respective hash value or “box”.  Finally, we implemented the self_collide function, in which we go through all the point masses in the same hash/box volume as a given point mass, and if the distance between each point mass and the input point mass is within 2 * the cloth’s thickness value, we consider it a collision and add an average correction vector (the average of the normalized vectors of the difference in both masses’ positions multiplied by the magnitude of 2 * thickness - the length of the difference vector) scaled down by simulation_steps. Afterwards, we call build_spatial_map and self_collide for all the point masses in the simulate function to build the hash table and correcting all self-collisions.

We ran into a lot of issues making this function as there were many segmentation faults caused due to the hash position returning NaN due to one of the point masses being returned as NaN. We later figured out that this was actually due to an error in our implementation of plane collision in part 3.

Part 5: Shaders

A shader program dictates how a model’s surface will be displayed in a given scene based on inputs of the external or internal conditions that control its physical or lighting properties. There are two types of shaders: vertex (.vert) and fragment (.frag) shaders. Vertex shaders are typically used to adjust the geometric properties of vertices within the mesh to demonstrate a physical texture (which is therefore used in displacement shaders that change the geometry of the mesh without explicitly changing the mesh itself). Fragment shaders, on the other hand, typically perform calculations from inputs such as the texture image or incoming light to display colors to display specific colors. These two work together by accurately displaying real-world properties for specific materials (i.e. for material like concrete, there are various grooves, ridges, and cavities that need both geometric distortions from vertex shaders as well as ways to dictate how lighting affects the darkness of shadows within a given cavity via fragment shaders, which is possible by using vertex shader computations as inputs to determine these colors.

Implementing each shader was fairly simple to implement as they involved either defined formulas or familiar implementations from previous projects (i.e. raytracing from project 3 for mirror shading and texture mapping from project 1 for texture shading).

Mirror Shader

For the mirror shading, we used a similar tactic to our previous raytracing algorithms from project 3. First, we calculated a vector wo as the normalized form of the camera position u_cam_pos - the vec3 x,y,z components of current position v_position. We then calculated a vector wi as the reflection of wo about the normal vector v_normal by evaluating it as wo - 2 * dot(v_normal.xyz, wo) * v_normal.xyz. Finally, as with the prior task, we assigned the final out_color to be the texture function call with parameters u_texture_cubemap (used for the reflection of the outside environment) and wi.
